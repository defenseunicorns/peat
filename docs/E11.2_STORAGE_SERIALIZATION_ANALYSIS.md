# E11.2 Storage Serialization Analysis

**Date:** 2025-11-13
**Status:** BLOCKING E11.2 Implementation
**Priority:** CRITICAL

## Problem Statement

E11.2 storage trait abstraction is blocked by fundamental serialization mismatch:

1. **Ditto requires JSON** for CRDT field-level merging
2. **Protobuf is binary** format, incompatible with JSON storage
3. **Trait abstraction** currently assumes opaque `Vec<u8>` (wrong for both!)

**Current approach loses ALL Ditto CRDT benefits** - we're storing base64 blobs instead of structured JSON.

## Current State Analysis

### What We're Doing NOW (WRONG)

```rust
// Current: Protobuf → Base64 → JSON blob
let bytes = summary.encode_to_vec();  // Binary protobuf
let base64_data = base64::encode(&bytes);  // Base64 string
let doc = json!({
    "squad_id": summary.squad_id,      // ✅ Indexed fields (good)
    "data": base64_data,                // ❌ Opaque blob (defeats CRDT!)
});
```

**Problems:**
- ❌ Ditto cannot merge `data` field at sub-field level
- ❌ Changes to nested protobuf fields trigger full blob replacement
- ❌ No delta sync - entire blob sent over network on any change
- ❌ Defeats bandwidth optimization (95%+ reduction claim invalid)
- ❌ **We're using Ditto as a dumb key-value store, not a CRDT engine**

### What We SHOULD Be Doing (CORRECT)

```rust
// Correct: Protobuf → Full JSON expansion
let json_doc = protobuf_to_json(&summary)?;  // Full field expansion
// Result:
{
    "_id": "squad-1",
    "squad_id": "squad-1",
    "leader_id": "node-123",
    "member_count": 5,
    "members": ["node-1", "node-2", "node-3"],  // ✅ Array for OR-Set CRDT
    "position": {                                // ✅ Nested object
        "latitude": 37.7749,
        "longitude": -122.4194
    },
    "capabilities": [...]                        // ✅ Full expansion
}
```

**Benefits:**
- ✅ Ditto merges `members` array with OR-Set semantics
- ✅ Ditto merges `position` fields with LWW-Register
- ✅ Delta sync sends only changed fields
- ✅ Query indexes work on all fields
- ✅ **Actual CRDT benefits as designed**

## Ditto CRDT Requirements

From `docs/Ditto-SDK-Integration-Notes.md`:

### CRDT Types Ditto Supports

1. **OR-Set** (Observed-Remove Set)
   - Use case: Cell membership, capability lists
   - Requires: JSON array `["item1", "item2"]`
   - Merging: Tracks additions/removals with causality

2. **LWW-Register** (Last-Write-Wins)
   - Use case: Position, fuel, health
   - Requires: JSON primitive (`number`, `string`, `boolean`)
   - Merging: Timestamp-based last-write-wins

3. **PN-Counter** (Positive-Negative Counter)
   - Use case: Fuel consumption tracking
   - Requires: JSON number
   - Merging: Increment/decrement operations

**All require structured JSON, NOT binary blobs!**

## Automerge Requirements

### Automerge Data Model

Automerge (E11.1 POC) stores documents as Automerge CRDT objects:

```rust
// Automerge stores native CRDT documents
let mut doc = Automerge::new();
doc.transact(|tx| {
    tx.put(ROOT, "squad_id", "squad-1")?;
    tx.put(ROOT, "leader_id", "node-123")?;
    tx.put(ROOT, "member_count", 5)?;
    // Full field expansion, similar to JSON
})?;
```

**Automerge binary format:**
- Stores CRDT metadata + field values
- Can save to bytes: `doc.save()` → `Vec<u8>`
- Can load from bytes: `Automerge::load(&bytes)?`
- **Optimized binary encoding, NOT JSON**

## Business Logic Data Access

### How CAP Protocol Uses Data

Coordinators need to:
1. **Query** - "Find all cells with >5 members"
2. **Filter** - "Get nodes within geohash prefix 9q8y"
3. **Aggregate** - "Count total capabilities across squad"
4. **Update** - "Change cell leader to node-456"

**Current trait interface:**
```rust
fn get(&self, doc_id: &str) -> Result<Option<Vec<u8>>>;  // ❌ Opaque bytes
```

**Business logic then must:**
```rust
let bytes = collection.get("cell-1")?.unwrap();
let cell = CellState::decode(&bytes[..])?;  // Protobuf decode
```

**Problem:** Caller handles serialization, not backend-specific!

## Solution Options

### Option 1: Typed Protobuf Trait (Recommended)

**Trait design:**
```rust
pub trait Collection: Send + Sync {
    /// Store a protobuf message
    fn upsert<M: prost::Message>(&self, doc_id: &str, msg: &M) -> Result<()>;

    /// Retrieve a protobuf message
    fn get<M: prost::Message + Default>(&self, doc_id: &str) -> Result<Option<M>>;

    /// Query with predicate on typed message
    fn find<M: prost::Message + Default>(
        &self,
        predicate: impl Fn(&M) -> bool,
    ) -> Result<Vec<(String, M)>>;
}
```

**Backend implementations:**
- **DittoCollection**: Protobuf → JSON (full expansion via serde) → Ditto
- **AutomergeCollection**: Protobuf → Automerge fields → save to bytes
- **RocksDB**: Protobuf → encode_to_vec → RocksDB (simple k/v)

**Benefits:**
- ✅ Typed interface - compiler checks message types
- ✅ Backend handles serialization format internally
- ✅ Ditto gets JSON for CRDT merging
- ✅ Automerge gets field-level access
- ✅ RocksDB gets efficient binary

**Challenges:**
- ❌ Requires `#[derive(Serialize, Deserialize)]` on protobufs
- ❌ Generic trait methods (complex trait object usage)
- ❌ May need wrapper types for serde compatibility

### Option 2: JSON Value Trait

**Trait design:**
```rust
pub trait Collection: Send + Sync {
    fn upsert(&self, doc_id: &str, value: serde_json::Value) -> Result<()>;
    fn get(&self, doc_id: &str) -> Result<Option<serde_json::Value>>;
}
```

**Backend implementations:**
- **Ditto**: JSON → Ditto (direct)
- **Automerge**: JSON → Automerge fields
- **RocksDB**: JSON → serialize to bytes → RocksDB

**Benefits:**
- ✅ Simple trait (no generics)
- ✅ Ditto gets native JSON
- ✅ Easy to inspect/debug

**Challenges:**
- ❌ Lose type safety (runtime errors)
- ❌ Business logic converts protobuf ↔ JSON manually
- ❌ Still need serde derives on protobufs

### Option 3: Hybrid Approach (Current + Metadata)

**Trait design:**
```rust
pub struct Document {
    pub data: Vec<u8>,                    // Binary protobuf
    pub indexed_fields: HashMap<String, serde_json::Value>,  // Queryable fields
}

pub trait Collection: Send + Sync {
    fn upsert(&self, doc_id: &str, doc: Document) -> Result<()>;
    fn get(&self, doc_id: &str) -> Result<Option<Document>>;
}
```

**Backend implementations:**
- **Ditto**: Extract `indexed_fields` + base64(`data`) → JSON
- **Automerge**: Ignore indexed_fields, use `data` directly
- **RocksDB**: Store `data` only

**Benefits:**
- ✅ Ditto gets queryable fields
- ✅ Binary protobuf preserved
- ✅ No serde derives needed

**Challenges:**
- ❌ **Still loses CRDT field-level merging** (defeats Ditto purpose!)
- ❌ Caller must extract indexed fields manually
- ❌ Duplicate data (in `data` and `indexed_fields`)

    fn upsert(&self, doc_id: &str, msg: &M) -> Result<()>;

#### Phase 3: Ditto Implementation (E11.2c)

```rust
impl<M> TypedCollection<M> for DittoCollection<M>
where
    M: prost::Message + Default + Serialize + DeserializeOwned,
{
    fn upsert(&self, doc_id: &str, msg: &M) -> Result<()> {
        // Protobuf → JSON (full expansion)
        let mut json = serde_json::to_value(msg)?;
        json["_id"] = serde_json::Value::String(doc_id.to_string());

        // Store in Ditto (gets full CRDT benefits)
        self.store.upsert(&self.collection_name, json).await?;
        Ok(())
    }

    fn get(&self, doc_id: &str) -> Result<Option<M>> {
        let results = self.store.query(
            &self.collection_name,
            &format!("_id == '{}'", doc_id),
        ).await?;

        if results.is_empty() {
            return Ok(None);
        }

        // JSON → Protobuf
        let msg: M = serde_json::from_value(results[0].clone())?;
        Ok(Some(msg))
    }
}
```

#### Phase 4: Automerge Implementation (E11.2 Week 2)

```rust
impl<M> TypedCollection<M> for AutomergeCollection<M>
where
    M: prost::Message + Default + Serialize + DeserializeOwned,
{
    fn upsert(&self, doc_id: &str, msg: &M) -> Result<()> {
        // Protobuf → JSON → Automerge fields
        let json = serde_json::to_value(msg)?;
        let mut doc = Automerge::new();

        doc.transact(|tx| {
            json_to_automerge(tx, ROOT, &json)?;
            Ok::<(), Error>(())
        })?;

        self.store.collection(&self.name).upsert(doc_id, doc)?;
        Ok(())
    }
}
```
